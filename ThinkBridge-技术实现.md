## §1 框架选择策略

AI 不是按关键词生硬匹配，而是先判断问题底层结构。

| 问题底层结构 | 主框架 | 验证框架 | 教学重点 |
|-------------|--------|----------|----------|
| 要不要做 | 决策矩阵、后悔最小化 | 逆向思维、安全边际 | 学会列选项和标准 |
| 为什么发生 | 五问法、鱼骨图 | 第一性原理、冰山模型 | 区分症状和根因 |
| 未来会怎样 | 情景规划、二阶思维 | 黑天鹅思维 | 不预测单一未来 |
| 该怎么做 | 金字塔原理、MECE | 二八定律 | 学会拆解行动路径 |
| 人际关系困局 | 换位思考、利益相关者地图 | 面子成本、中庸思维 | 看见对方利益和台阶 |
| 职业/人生选择 | 后悔最小化、反脆弱 | 安全边际、成长型思维 | 平衡长期价值和短期风险 |
| 创新/突破 | 第一性原理、SCAMPER | 类比推理、逆向思维 | 打破默认假设 |

## §2 框架库配置化

32 个框架的定义不能硬编码进 Prompt。采用配置化结构，第一版在单文件 HTML 内部用 JS 常量对象模拟，结构上保持可拆分为独立 JSON 文件。

每个框架的配置结构：

```json
{
  "id": "decision_matrix",
  "name": "决策矩阵",
  "category": "决策类",
  "summary": "用于多个选项之间的结构化比较。",
  "solves": "多选项决策，不知道该选哪个。",
  "thinking_order": ["列选项", "列标准", "设权重", "打分", "复盘"],
  "teaching_points": [...],
  "self_questions": [...],
  "common_misuses": [...]
}
```

现实约束过滤器同理，配置化存储触发场景和检查项。

Step 1 Router 使用框架简表（名称、类别、一句话用途），Step 2 Analyzer 使用被选中框架的完整定义。这样可以保证每次只注入必要信息，降低上下文长度。

## §3 AI 工作流（两步 Pipeline）

如果一次请求塞入全部 32 个框架定义、现实约束规则、教学模板，容易出现上下文过长、注意力稀释、输出格式混乱等问题。因此采用分层调用：诊断和教学分离。

### Step 1：路由与诊断（Router）

目标：只负责判断问题类型和选择框架，不生成长报告。

输入：
- 用户问题
- 框架库简表（id、名称、类别、一句话用途）
- 现实约束触发规则简表

输出：强制 JSON。

```json
{
  "status": "ready",
  "needs_clarification": false,
  "clarifying_questions": [],
  "problem_type": "职业/人生选择",
  "problem_subtype": "多选项决策 + 风险控制",
  "selected_frameworks": ["决策矩阵", "后悔最小化", "安全边际"],
  "validation_frameworks": ["逆向思维"],
  "reality_filters": ["关系思维", "面子成本"],
  "reason": "该问题涉及多个选项比较、长期后悔和短期风险控制。"
}
```

如果信息不足，`status` 返回 `"need_more_info"`，前端展示追问并停止后续调用。

**JSON 解析兜底策略：**

LLM 输出的 JSON 可能被 markdown 代码块包裹、含尾随逗号或混入解释文字。前端必须按以下顺序尝试解析：

```text
1. 直接 JSON.parse(response)
2. 提取 ```json ... ``` 代码块内容，再 JSON.parse
3. 正则提取第一个 { ... } 对象，再 JSON.parse
4. 发起一次"修复重试"：Prompt 只包含原始输出 + "请修复以上 JSON，只返回合法 JSON，不要其他内容"
5. 仍失败 → 降级：使用默认框架组合，提示用户"框架选择遇到问题，已使用默认分析方案"
```

降级默认组合：按问题大类预设，例如"要不要做"类默认用「决策矩阵 + 后悔最小化」，「为什么发生」类默认用「五问法 + 冰山模型」。

### Step 2：分析与教学（Analyzer）

目标：只针对 Step 1 选出的框架进行深度教学型分析。

输入：
- 用户问题
- Step 1 的 JSON 诊断结果
- 被选中框架的详细定义（含思考顺序、教学要点、自我提问模板、常见误区）
- 选中现实约束过滤器的详细规则
- 教学型输出模板

输出：流式 Markdown。

优势：
- 每次只注入必要框架，降低上下文长度
- 提高模型注意力，输出更稳定
- 方便以后替换模型或增加框架而不需要重写 Prompt

### 教学型框架分析结构

每个框架按以下结构输出（同 Step 2）：

```markdown
## 框架：决策矩阵

### 1. 这个框架解决什么问题
### 2. 它的思考顺序
### 3. 为什么要这样思考
### 4. 代入你的问题示范
### 5. 你以后怎么自己用
### 6. 常见误区
```

### 交叉验证与综合报告

Step 2 输出中自然包含交叉验证（框架间共识、矛盾、修正）和综合建议（方向判断、风险提醒、行动清单、独立思考和模板、练习题）。

## §5 Prompt 设计（两步 Pipeline）

### Step 1：Router Prompt

```text
你是 ThinkBridge 的问题诊断器。

你的任务是根据用户问题，判断问题类型并选择合适的思维框架。

规则：
1. 先判断问题底层结构（decision / root-cause / future / action-plan / interpersonal / career / innovation）
2. 从框架库中选择 1-3 个主框架 + 1-2 个验证框架
3. 根据问题涉及的中国现实场景，选择必要的现实约束过滤器
4. 如果信息不足，返回 status="need_more_info" 和 1-3 个追问
5. 必须返回 JSON，不要输出其他内容

框架库简表：
（32 个框架的 id、名称、类别、solves 字段）

现实约束触发规则：
（触发场景对应关系）

输出格式：
{"status": "ready"|"need_more_info", "needs_clarification": bool, ...}
```

### Step 2：Analyzer Prompt

```text
你是 ThinkBridge，一个 AI 思维框架教练。

你的目标不是只给用户答案，而是教会用户如何使用思维框架。

当用户提出问题时，你需要：
1. 判断这个问题本质属于哪类问题
2. 选择 1-3 个合适的思维框架
3. 解释为什么选择这些框架
4. 解释每个框架的底层逻辑
5. 展示这个框架的思考步骤
6. 每一步都说明"为什么要这样思考"
7. 把用户的问题代入框架完整示范
8. 说明这个框架的常见误区
9. 总结用户以后可以复用的自我提问模板
10. 最后给 1-3 个练习题，帮助用户迁移掌握

你不是替用户做决定，而是帮助用户看清问题结构、关键变量、风险和可行动方案。

当前选中的框架：
{{Step 1 输出的 selected_frameworks 详细定义}}

当前现实约束：
{{Step 1 输出的 reality_filters 详细规则}}

注意：
- 不要只给结论
- 不要堆概念
- 不要展示内部推理链
- 用简单中文解释
- 明确说明每一步为什么要这样思考
- 对医疗、法律、金融投资问题，不给确定性建议
- 输出使用 Markdown
```

### 用户 Prompt

```text
请用"学习模式"分析下面这个问题：

{{用户输入}}

请按以下结构输出：
1. 问题理解
2. 问题类型诊断
3. 为什么选择这些框架
4. 每个框架的思考方式、步骤、为什么这样思考
5. 把我的问题代入框架示范
6. 现实约束检查
7. 交叉验证
8. 综合建议
9. 我以后如何自己使用这个框架
10. 练习题
```

## §6 PWA 离线策略

ThinkBridge 的核心功能（AI 分析）必须联网，但作为 PWA 需要定义离线行为。

### 离线可用

- 浏览框架库（数据来自 JS 常量，不依赖网络）
- 查看历史报告（来自 localForage 本地存储）
- 搜索和筛选框架

### 离线不可用

- 提交新问题进行分析
- 获取流式响应

### 离线体验

```text
检测到网络断开时：
- 分析页的输入框和提交按钮置灰
- 显示提示："当前离线，分析功能暂不可用。你可以浏览框架库或查看历史报告。"
- 框架库页和历史页正常可用

网络恢复时：
- 自动恢复分析功能
- 不需要用户手动刷新
```

### Service Worker 缓存策略

```text
Cache First       ：HTML、CSS（Tailwind CDN）、JS 库（Alpine.js、marked.js、localForage CDN）
Network First     ：分析页 HTML（保证拿到最新版本）
```

离线状态下不缓存用户输入的问题文本到 Service Worker Cache，避免敏感数据残留在缓存中。

## §7 技术选型

| 层 | 选型 | 原因 |
|----|------|------|
| 前端 | 单文件 HTML + Tailwind CDN | 零构建、手机优先、方便本地打开 |
| AI | DeepSeek API | 成本低，适合个人使用 |
| AI 链路 | 两步 Pipeline（Router + Analyzer） | 避免上下文过载，提高输出稳定性 |
| 调用方式 | 浏览器 fetch DeepSeek API | 本地自用，先不加后端；若 CORS 不通则加 Cloudflare Worker 代理 |
| 流式 | fetch stream | 让用户看到分析进度和报告逐步生成 |
| 轻量存储 | localStorage | 保存 API Key、设置 |
| 长文本存储 | localForage / IndexedDB | 保存历史报告，避免 5MB localStorage 限制 |
| UI 状态 | Alpine.js（CDN 引入） | 轻量、无构建、适合单文件 HTML |
| Markdown | marked.js（CDN 引入） | M1 纯文本，M2 按块增量渲染 |
| 运行 | 本地浏览器 | 暂不上线 |
| PWA | manifest + service worker | 后续支持添加到桌面 |

## §8 本地数据结构

### 轻量设置（localStorage）

API Key：
```text
thinkbridge_api_key
```

设置：
```json
{
  "mode": "learning",
  "model": "deepseek-chat",
  "temperature": 0.7,
  "stream": true,
  "language": "zh-CN"
}
```

### 历史报告（localForage / IndexedDB）

localStorage 通常只有约 5MB，长 Markdown 报告几十条后可能触发 QuotaExceededError，因此使用 localForage 存储。

```json
[
  {
    "id": "timestamp",
    "question": "我该不该跳槽？",
    "mode": "learning",
    "frameworks": ["决策矩阵", "后悔最小化", "安全边际"],
    "answer": "完整 Markdown 报告",
    "diagnosis": { "problem_type": "...", ... },
    "createdAt": "2026-05-23 20:30:00"
  }
]
```

## §9 错误处理

| 场景 | 提示 |
|------|------|
| 没有 API Key | 请先填写 DeepSeek API Key |
| API Key 错误 | 认证失败，请检查 API Key 是否正确 |
| 网络失败 | 请求失败，请检查网络连接 |
| 额度不足 | DeepSeek 返回额度错误，请检查账户余额 |
| 输入太短 | 问题信息较少，建议补充背景、目标和限制条件 |
| 输出中断 | 生成被中断，可以重试或复制已生成内容 |
| Step 1 JSON 解析失败 | 自动尝试修复；多次失败后使用默认框架降级分析 |
| Step 2 流式中断 | 保留已生成内容，提供"继续生成"和"复制已有内容"按钮 |
| localStorage 不可用 | 当前浏览器无法保存设置，请检查隐私设置 |
| localForage 不可用 | 降级使用 localStorage，若仍不可用则本次不保存历史 |

## §10 实现计划

### 开发原则

```text
1. 模块化、低耦合：每个模块独立可测，不依赖未完成的部分
2. 小步前进：每次只做一个任务，做完立即验证
3. 验证通过才继续：验证失败不回滚，就地修复
4. 每个模块完成后保存为 checkpoint（git commit 或文件快照）
```

每个任务包含三要素：**做什么 → 怎么验证 → 验证通过后的下一步**。

---

### Milestone 1：跑通核心 AI 链路

#### 模块 A：静态骨架（可独立完成，不依赖 API）

| # | 做什么 | 怎么验证 | 依赖 |
|---|--------|----------|------|
| A1 | 极简 HTML：输入框、API Key 设置、输出区域（纯 `<pre>` 显示） | 浏览器打开页面，输入文字，点击按钮后能在输出区看到 echo | 无 |
| A2 | 框架配置 JS 常量：32 个框架的简表 + 3-5 个框架的完整定义 | 控制台 `console.log(FRAMEWORKS)` 能打印出数据结构 | A1 |
| A3 | 现实约束配置 JS 常量：触发场景 + 检查项 | 同上验证 | A1 |
| A4 | 框架库浏览页（纯前端）：分类筛选、框架卡片、详情弹出、"用此分析"按钮 | 点击每个框架卡片能看到详情，"用此分析"按钮能跳回分析页并预填文本 | A2 |

模块 A 完成标志：页面静态可用，框架数据可浏览，但还不能调 AI。

#### 模块 B：API 通道验证（独立小步）

| # | 做什么 | 怎么验证 | 依赖 |
|---|--------|----------|------|
| B1 | CORS 测试：fetch DeepSeek API，用最小 Prompt（"回复：hello"） | 控制台看到 200 响应和完整 JSON body | A1 |
| B2 | 若 CORS 不通：部署 Cloudflare Worker 代理，修改 fetch URL | 同上验证通过 | B1 失败时 |

模块 B 完成标志：浏览器能成功调用 DeepSeek API 并收到响应。

#### 模块 C：Step 1 Router（依赖 B）

| # | 做什么 | 怎么验证 | 依赖 |
|---|--------|----------|------|
| C1 | Router 调用：发送用户问题 + 框架简表，接收 JSON | 输入"我该不该跳槽"，控制台看到合法 JSON，`status` 字段存在 | A2, B1 |
| C2 | JSON 解析兜底：实现 5 级解析策略 | 故意构造畸形 JSON 字符串测试，最终拿到合法对象 | C1 |
| C3 | 信息不足处理：`status=need_more_info` 时展示追问，停止后续调用 | 输入极短问题（如"怎么办"），页面显示追问而非报告 | C1 |
| C4 | JSON 解析 5 次全失败时：降级默认框架，显示提示 | Mock 5 次失败，确认降级逻辑触发 | C2 |

模块 C 完成标志：输入问题 → 稳定拿到 `{status, selected_frameworks, ...}` JSON。

#### 模块 D：Step 2 Analyzer（依赖 C）

| # | 做什么 | 怎么验证 | 依赖 |
|---|--------|----------|------|
| D1 | Analyzer 调用：注入 Router JSON + 选中框架详情 + 模板，接收 Markdown | 输出内容包含"## 1. 问题理解""## 框架"等模板结构 | C1, A2 |
| D2 | 端到端串联：输入问题 → Router → Analyzer → 完整报告 | 完整走一遍，报告包含问题诊断 + 框架分析 + 自我提问模板 | D1 |

模块 D 完成标志：输入真实问题 → 完整教学型报告输出到页面。

**M1 总验收**：能稳定选出合适框架，能在信息不足时追问，能输出教学型分析（不频繁截断）。暂不做漂亮 UI、历史记录、PWA。

### Milestone 2：数据与流式体验

| # | 做什么 | 怎么验证 | 依赖 |

## §11 学习效果度量（V2 设计，第一版只做数据预留）

产品核心目标是"逐渐减少对 AI 的依赖"，需要可度量的方式。第一版不做完整实现，但数据结构预留字段。

### V2 功能设计

**每次分析后收集反馈：**

```text
分析报告底部显示两个按钮：
[有帮助 ✓] [不太行 ✗]

点击后记录到历史条目中，不弹窗不打断。
```

**框架学习记录：**

```json
{
  "frameworks_learned": {
    "decision_matrix": {
      "times_used": 5,
      "last_used": "2026-05-23",
      "user_rated_helpful": 4
    }
  }
}
```

**"你先试试"提示：**

```text
当用户提交新问题，Router 诊断后发现：
该问题类型与某次历史分析相同，且用户已使用过相关框架 ≥ 3 次

则在分析前插入提示：
"你之前用过【决策矩阵】分析类似问题（3次）。
要不要先自己试一遍，再看 AI 的分析？"
→ [ 我自己试试 ] [ 直接看分析 ]
```

**第一版数据预留：**

历史记录 JSON 中预留以下字段（暂不实现 UI）：

```json
{
  "feedback": null,
  "user_tried_first": false
}
```

并在框架配置中预留 `times_used` 计数器字段，供 V2 读取。
